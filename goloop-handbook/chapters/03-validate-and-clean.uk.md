# 03. Валідація й чистка вводу

**Задача.** Форма реєстрації приходить повна людського безладу: зайві пробіли,
змішаний регістр, невидимий символ, вставлений із чату, номер телефону з тире й
дужками. Відхилити те, що справді невалідне, і тихо канонізувати те, що просто
неохайне - перш ніж будь-що з цього дійде до бази.

**Модулі.** [`is`](https://github.com/goloop/is) відповідає на *«чи це
валідне?»* предикатами лише для читання; [`norm`](https://github.com/goloop/norm)
відповідає на *«зроби це канонічною формою»* чисткою й згортанням.

**Рецепт.** [`recipes/003-validate-and-clean`](../recipes/003-validate-and-clean/)

## Ідея

Обробка вводу - це дві роботи, і їх змішування породжує баги. Одні поля мусять
бути *точно* валідними (email або є адресою, або ні). Інші поля мають бути
*поблажливими* (ім'я з хвостовим пробілом - нормально, просто обріж). `is` - це
половина лише для читання: він ніколи не змінює значення, лише повідомляє.
`norm` - половина запису: він повертає почищене значення і, де доречно, чи воно
валідне.

Чистка вільного текстового поля - це один виклик:

```go
name := norm.Clean(f.Name) // прибрати невидимі/контрол-байти, стиснути пробіли, тримити
if name == "" {
	return Account{}, ErrEmptyName
}
```

`norm.Clean` прибирає невидимі й контрольні символи (зокрема той вставлений
невидимий пробіл), стискає кожну серію пробілів в один і тримить краї. Поле, що
почистилося в порожнечу, було порожнім.

Email - це ідентичність, тож згорни його регістр:

```go
email, ok := norm.EmailFold(f.Email) // валідація + весь адрес у нижній регістр
if !ok {
	return Account{}, ErrInvalidEmail
}
```

`norm.EmailFold` валідує й переводить у нижній регістр увесь адрес, тож
`Ada@Example.COM` і `ada@example.com` - той самий акаунт. (`norm.Email` теж є й
зберігає регістр local-part для точної доставки - бери `EmailFold`, коли адрес є
ключем пошуку.)

Необов'язкове поле чистять, коли воно є, і відкидають, коли ні:

```go
phone := ""
if f.Phone != "" {
	if e164, ok := norm.E164(f.Phone); ok {
		phone = e164 // +14155550132
	}
}
```

## Звіт виконання

Чотири форми через `process`, далі read-only-предикати `is` напряму:

```text
$ go test ./...
ok  	goloop.one/handbook/003-validate-and-clean	0.004s

$ go run .
form 1: name="Ada Lovelace" email="ada@example.com" phone="+14155550132"
form 2: name="Grace Hopper" email="grace@example.com" phone=""
form 3: rejected: a name is required
form 4: rejected: a valid email is required
---
is.Email("a@b.co") = true
is.Numeric("12345") = true
```

Форма 1 прийшла як `"  Ada​ Lovelace "` (з невидимим пробілом після «Ada»),
`"  Ada@Example.COM "` і `"+1 (415) 555-0132"`; вона вийшла чистою, згорнутою і
в E.164. Ім'я форми 3 - самотній таб, що чиститься в порожнечу, тож її
відхилено. Email форми 4 - не адреса, тож її відхилено. Ніщо недовірене не
прослизнуло, і ніщо просто неохайне не викинуто.

## Що ти дізнався

- Розділяй роботу: **валідуй** те, що має бути точним (`is`), **чисть** те, що
  має бути поблажливим (`norm`).
- `norm.Clean` - безпечна one-call чистка будь-якого вільного текстового поля.
- `norm.EmailFold` дає case-insensitive ключ-ідентичність; `norm.Email`
  зберігає регістр для доставки.
- Необов'язкові поля (`norm.E164` тощо) повертають `(значення, ok)`, тож можна
  чистити-якщо-є, не відхиляючи всю форму.

Ти дійшов кінця Частини I: конфігурація всередину, запити змаршрутизовано, ввід
почищено. Частина II переходить до збереження цих чистих даних у PostgreSQL і
запитів до мовної моделі про них.
